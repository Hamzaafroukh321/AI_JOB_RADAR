# Material Assumptions

| ID | Assumption | Reason | Impact | Validation / future decision | Status |
|---|---|---|---|---|---|
| A-001 | English is the initial interface language. | Specification Section 42 safe default. | UI copy and locale default are English; architecture remains internationalization-ready. | Confirm during Phase 1 onboarding design. | Accepted for Phase 0 |
| A-002 | Browser authentication uses a server-side Spring Session stored in PostgreSQL with an HTTP-only cookie. | Best fit for a personal browser application and avoids browser token storage. | Requires CSRF protection and shared session persistence; supports later horizontal scaling. | Revisit only if deployment topology changes materially. | Accepted |
| A-003 | Java 21, Spring Boot 4.0.x, Angular 21, PrimeNG 21, and PrimeFlex 4 are pinned. | They are mutually compatible supported stable lines available during initialization; Java and Node runtimes are installed locally. | Establishes build/runtime baselines and requires Maven Wrapper because Maven is absent globally. | Review supported releases before future major upgrades; record upgrades in ADRs/changelog. | Accepted |
| A-004 | Development seed login is opt-in and requires environment-provided email/password; there is no baked-in credential. | Prevents a known default account from leaking into production. | First local login requires explicit `.env` configuration. | Validate with seed idempotency and production-disable tests. | Accepted |
| A-005 | PostgreSQL, MinIO, and Mailpit run locally in Docker; backend/frontend may run on the host or in containers. | Matches the specified happy path and supports deterministic infrastructure. | Docker Desktop is required for integration and smoke validation. | Prove service health in the Phase 0 acceptance run. | Accepted |
| A-006 | Candidate languages, work authorization, sponsorship, relocation, and other unknown profile facts remain unseeded. | The specification forbids inference. | No candidate/profile production seed exists in Phase 0. | Collect explicit user confirmation in Phase 1. | Accepted |
| A-007 | Contract/freelance is allowed, strict worldwide-remote filtering is required, email sending is disabled, and in-app notifications are the future default. | Specification Section 42 safe defaults. | Recorded now only; no later-phase feature is implemented. | Validate in the relevant later phases. | Deferred decision |
| A-008 | The provided GitHub remote is empty, so Git metadata can be initialized in the current workspace and the remote recorded without overwriting files. | Read-only `git ls-remote` returned no refs and the workspace has no `.git`. | Establishes a normal working tree while preserving both supplied files. | Confirm with `git status` and remote listing. | Accepted |
| A-009 | Phase 1 performs deterministic PDF/DOCX extraction and proposal creation; AI-assisted resume mapping may replace or augment that boundary later. | The delivery plan introduces the Groq provider in Phase 3, while Phase 1 must remain independently runnable and testable. | Resume upload, provenance, review, and verification work without a network key. | Revisit the proposal implementation after the Phase 3 provider is accepted. | Accepted for Phase 1 |
| A-010 | Candidate seed facts from the specification import as `PROPOSED`, despite being described as resume-derived verified facts. | The same specification requires onboarding review and forbids silent inference. | The user must explicitly verify each imported fact before generation can query it. | Validate through seed-import and verified-query tests. | Accepted |
| A-011 | Adzuna is the initial documented query-API adapter; it remains disabled until supported-country configuration, attribution, terms approval, and runtime credential references are supplied. | The specification explicitly lists Adzuna and requires one query API connector in Phase 2. | Contract parsing is testable offline and no key is stored in source configuration. | Validate a live source only outside automated tests after the operator approves terms and supplies fresh credentials. | Accepted for Phase 2 |
| A-012 | Phase 2 manual import guarantees pasted-description ingestion; safe URL fetching may return a reason requiring paste instead of attempting browser-like access. | Many boards disallow automated access and manual paste is the specified fallback. | Original URLs remain attached, but no authenticated or evasive fetch is attempted. | Extend only for explicitly approved public URLs passing the SSRF policy. | Accepted |
| A-013 | Phase 3 uses deterministic analysis as the available baseline and invokes Groq only when explicitly enabled with a runtime key. | Search and classification must remain runnable offline, and the available task environment does not inherit the user’s separate shell secret. | Model enrichment can be retried later; no live model call is needed for acceptance. | Validate the adapter separately with a fresh operator-controlled key outside automated tests. | Accepted for Phase 3 |
| A-014 | Strict worldwide/Morocco sections use deterministic geographic evidence even when model output is available. | “Remote” alone must not become worldwide and eligibility safety should not depend on probabilistic output. | Ambiguous remote roles remain visible in All AI Jobs with an explicit unknown explanation. | Revisit rule packs with production feedback; never relax without evidence tests. | Accepted |
| A-015 | Phase 4 computes matches synchronously on explicit access and records recomputation requests for profile/job changes. | Docker-backed asynchronous infrastructure is unavailable during this continuation. | Results remain correct on access while the queue provides a later worker boundary. | Add scheduled queue draining during production hardening. | Accepted for Phase 4 |
A-016. Tailored resume generation defaults to an offline deterministic planner; optional Groq
planning is not required for document availability and never bypasses fact validation.

A-017. The five initial resume variants are AI Engineer, GenAI/RAG, Java/Angular, AI
Training/Evaluation, and Remote Contract. English two-page ATS output is the safe default.

A-018. This installation is optimized for one junior software/AI engineer. The default job search
requires title-level AI/software relevance and excludes explicit senior leadership and unrelated
commercial roles. Stricter classified-junior and explicit junior/entry-signal filters remain
available when precision is more important than recall.
